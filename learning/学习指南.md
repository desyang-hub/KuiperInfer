### Day 2：Tensor 张量类

**目标**：深入理解张量这个推理框架的基石。

**阅读文件**：
- `include/data/tensor.hpp`
- `source/data/tensor.cpp`

**核心知识点**：

1. **Tensor 的内部存储**

   `Tensor<T>` 内部使用 Armadillo 的 `arma::Cube<T>` 作为底层存储：

   ```cpp
   template<typename T>
   class Tensor {
       std::shared_ptr<arma::Cube<T>> data_;  // 实际数据
       std::vector<int32_t> shapes_;           // 逻辑形状 [channels, rows, cols]
   };
   ```

   **关键设计**：虽然叫 `Cube`（3D），但框架智能地处理维度降阶：
   - 创建 `[1, 1, N]` → 实际存储为 1D
   - 创建 `[1, H, W]` → 实际存储为 2D
   - 创建 `[C, H, W]` → 存储为 3D

2. **常用操作**

   | 方法 | 作用 |
   |------|------|
   | `Reshape()` | 改变张量形状（支持行优先/列优先） |
   | `Flatten()` | 展平为一维 |
   | `Fill()` | 按指定值填充 |
   | `Padding()` | 空间维度填充（卷积预处理） |
   | `Transform()` | 逐元素应用 lambda 函数 |
   | `slice(i)` | 获取第 i 个 channel 的矩阵切片 |

3. **内存管理**

   Tensor 支持两种模式：
   - **所有权模式**：Tensor 拥有数据，析构时自动释放
   - **非所有权模式**：Tensor 包装外部指针，不管理内存

   ```cpp
   // 非所有权：ptr 是外部数据，false=不复制，true=管理生命周期
   arma::Cube<T>(ptr, h, w, c, false, true)
   ```

**实操练习**：

1. 阅读 `test/test_data/` 下的测试文件，理解 Tensor 的各种操作
2. 尝试在测试中添加一个新的 Tensor 操作测试

**思考题**：
- Armadillo 内部是**列优先（column-major）**存储，为什么深度学习框架通常用**行优先（row-major）**？Tensor 如何桥接这两种方式？
- `shared_ptr<Tensor<T>>`（别名 `sftensor`）为什么用智能指针而非裸指针？（多所有者共享、零拷贝传播）

## 第二阶段：算子（Layer）系统设计（Day 4-7）

### Day 4：算子注册工厂模式

**目标**：掌握算子的统一管理和自动注册机制。

**阅读文件**：
- `include/layer/abstract/layer.hpp`
- `include/layer/abstract/layer_factory.hpp`
- `source/layer/abstract/layer_factory.cpp`

**核心知识点**：

1. **Layer 抽象基类**

   ```cpp
   template<typename T>
   class Layer {
   public:
       virtual ~Layer() = default;
       virtual StatusCode Forward() = 0;                    // 使用关联的 RuntimeOperator 中的数据
       virtual StatusCode Forward(const std::vector<sftensor>& inputs,
                                  std::vector<sftensor>& outputs) = 0;  // 显式传参
   };
   ```

   两个 `Forward()` 重载：
   - 无参版：从关联的 `RuntimeOperator` 中读取输入输出数据
   - 显式传参版：直接传入输出和输出 Tensor，用于单元测试和 Demo

2. **工厂注册模式（重点！）**

   这是整个框架最精妙的设计之一。**每个算子类型在程序启动时自动注册到自己**：

   ```cpp
   // LayerRegisterer 维护一个全局注册表
   class LayerRegisterer {
       static std::map<std::string, Creator>& GetRegistry() { ... }
   public:
       static StatusCode RegisterCreator(const std::string& type, Creator creator);
       static StatusCode CreateLayer(const std::string& type, RuntimeOperator& op, Layer<float>& layer);
   };

   // LayerRegistererWrapper 在构造时自动注册
   class LayerRegistererWrapper {
   public:
       LayerRegistererWrapper(const std::string& type, Creator creator) {
           LayerRegisterer::RegisterCreator(type, creator);
       }
   };
   ```

   **算子自注册**（以 ReLU 为例）：
   ```cpp
   // relu.cpp 中的全局变量
   LayerRegistererWrapper kReluCreateInstance(ReluLayer::CreateInstance, "nn.ReLU");
   ```

   当可执行文件加载时，`kReluCreateInstance` 这个全局变量被构造，构造函数调用 `RegisterCreator`，将 `ReluLayer::CreateInstance` 函数指针注册到全局 map 中，键名为 `"nn.ReLU"`（即 PNNX 模型中算子的类型名）。

   **运行时创建算子**：
   ```cpp
   // 给定类型字符串 "nn.ReLU"，从注册表中找到对应的工厂函数
   LayerRegisterer::CreateLayer("nn.ReLU", runtimeOp, layer);
   ```

3. **类层次结构**

   ```
   Layer<float>                     // 抽象基类
   ├── NonParamLayer (type alias)   // 无权重算子（池化、激活等）
   ├── ParamLayer                   // 有权重算子（卷积、全连接等）
   │   ├── BaseConvolutionLayer     // 卷积/反卷积抽象基类
   │   │   ├── ConvolutionLayer
   │   │   └── DeconvolutionLayer
   │   └── LinearLayer
   └── ActivationLayer              // 激活函数基类
   ```

**画图理解**：

```
PNNX 模型文件中有 "nn.ReLU" 类型字符串
        ↓
RuntimeGraph::Build() 调用 LayerRegisterer::CreateLayer("nn.ReLU", ...)
        ↓
注册表中查找 "nn.ReLU" → 找到 ReluLayer::CreateInstance
        ↓
调用 CreateInstance(runtimeOp, layer) → 创建 ReluLayer 实例
        ↓
Forward() 执行时调用 ReluLayer 的实现
```

**思考题**：
- 为什么用全局变量构造函数来注册，而不是手动在一个地方注册所有算子？（新增算子无需修改注册中心，符合开闭原则）
- `Layer<float>` 有一个 `weak_ptr<RuntimeOperator>`，为什么不用 `shared_ptr`？（避免循环引用导致内存泄漏）

### Day 5：卷积算子深度分析

**目标**：理解框架中最核心的算子——卷积的完整实现。

**阅读文件**：
- `source/layer/details/base_convolution.cpp`
- `source/layer/details/convolution.cpp`
- `include/layer/details/base_convolution.hpp`
- `include/layer/details/convolution.hpp`

**核心知识点**：

1. **卷积数学公式回顾**

   对于输入 $I$、卷积核 $K$、偏置 $b$：
   $$O_{n,c} = \sum_{ic=0}^{C_{in}/G-1} K_{c,ic} * I_{n,ic} + b_c$$

   其中 $G$ 是 group 数，$*$ 表示二维互相关操作。

2. **im2col + GEMM 优化（重中之重！）**

   这是几乎所有高性能推理框架（Caffe、NCNN、TensorRT）都使用的卷积加速技术。

   **核心思想**：把卷积操作转换为矩阵乘法。

   ```
   原始卷积：输入 [C,H,W] * 卷积核 [K,C/k,kh,kw] → 输出 [K,H',W']
   
   im2col 转换后：
   输入矩阵 [C*kh*kw, H'*W']  ×  卷积核矩阵 [K, C*kh*kw]  →  输出矩阵 [K, H'*W']
   ```

   **具体步骤**：

   **(1) 权重预处理** (`InitIm2ColWeight`)
   ```cpp
   // 将 N-D 卷积核展平为 2D 矩阵
   // 每个卷积核 [out_ch, in_ch/groups, kh, kw] → 展平为行向量 [in_ch/groups * kh * kw]
   // 所有卷积核组成矩阵 [out_ch, in_ch/groups * kh * kw]
   ```

   **(2) im2col 变换** (`ConvIm2Col`)
   ```cpp
   // 将输入的空间区域 "展开" 为列
   // 对于输出的每个位置 (h', w')，把对应感受野内的所有值拼成一个列向量
   // 结果矩阵的每一列 = 一个输出位置的感受野展平
   ```

   **(3) GEMM 乘法**
   ```cpp
   // 直接用 Armadillo 做矩阵乘法（底层调用 OpenBLAS 的 sgemm）
   output_matrix = kernel_matrix * input_matrix;
   ```

   **(4) 加偏置**
   ```cpp
   output_matrix.each_row() += bias_row;
   ```

3. **1×1 卷积特化优化**

   当卷积核为 1×1 且无 padding 时，im2col 是多余的操作：
   ```cpp
   // 直接将输入 [C, H, W] 视为矩阵 [C, H*W]
   // 卷积核 [K, C, 1, 1] 视为矩阵 [K, C]
   // output = kernel @ input  →  [K, H*W]
   ```

4. **Group 卷积支持**

   ```cpp
   // 将输入通道和卷积核按 group 划分
   // 每个 group 独立做卷积，最后拼接结果
   for (int group = 0; group < groups_; group++) {
       // 对每个 group 的输入通道子集执行卷积
   }
   ```

5. **BaseConvolutionLayer 模板方法模式**

   ```cpp
   class BaseConvolutionLayer {
   public:
       Forward() {
           Check();                        // ① 验证输入
           InitIm2ColWeight();             // ② 预处理权重
           for (batch) {                   // ③ 遍历 batch
               for (group) {               // ④ 遍历 group
                   ComputeOutput();        // ⑤ 子类实现的具体卷积计算
               }
           }
       }
   protected:
       virtual void ComputeOutput() = 0;   // 卷积/反卷积分别实现
       virtual void ComputeOutputSize() = 0;
   };
   ```

**实操练习**：

1. 手绘 im2col 的过程：用一个 [3, 4, 4] 输入和 [2, 3, 2, 2] 卷积核，手动展开
2. 阅读 `test/test_layer/` 中卷积相关的测试
3. 对比 `convolution.cpp` 和 `deconvolution.cpp` 的区别

**思考题**：
- im2col 把卷积转成了 GEMM，为什么这样更快？（BLAS 库经过数十年高度优化，利用了 CPU 缓存、SIMD、多线程等一切手段）
- im2col 的代价是什么？（额外的内存分配和数据复制）

### Day 6：池化与归一化算子

**目标**：学习池化和归一化算子的实现。

**阅读文件**：
- `source/layer/details/maxpooling.cpp`
- `source/layer/details/adaptive_avgpooling.cpp`
- `source/layer/details/batchnorm2d.cpp`

**核心知识点**：

1. **MaxPooling**
   - 滑动窗口取最大值
   - Padding 处理：手动检查边界，越界处用 `float::lowest()` 填充
   - OpenMP 在 batch 维度并行

   ```cpp
   #pragma omp parallel for num_threads(batch)
   for (int b = 0; b < batch; b++) {
       for (each channel) {
           for (each output position) {
               for (each element in window) {
                   // 取最大值
               }
           }
       }
   }
   ```

2. **AdaptiveAvgPooling**
   - **自适应**：根据输入尺寸动态计算 stride 和 kernel_size
   ```
   stride_h = floor(input_h / output_h)
   kernel_h = input_h - (output_h - 1) * stride_h
   ```
   - 直接累加窗口内所有值，然后除以窗口大小
   - 使用 `raw_ptr()` 直接操作内存，避免 Armadillo 开销

3. **BatchNorm2D（推理模式）**

   推理时 BatchNorm 简化为逐 channel 的仿射变换：
   $$output[i] = \frac{input[i] - \mu_i}{\sqrt{\sigma_i^2 + \epsilon}} \times \gamma_i + \beta_i$$

   其中 $\mu$（running_mean）、$\sigma^2$（running_var）、$\gamma$（affine_weight）、$\beta$（affine_bias）都是预计算好的参数。

**实操练习**：

- 阅读 `test/test_layer/test_maxpooling.cpp` 和 `test_adaptive_avgpooling.cpp`
- 理解 AdaptiveAvgPooling 在不同输入尺寸下的行为

### Day 7：激活函数与 SIMD 优化

**目标**：学习激活函数的 SIMD 向量化实现。

**阅读文件**：
- `source/layer/details/activation.cpp`
- `source/layer/details/simd.cpp`（或 `simd.hpp`）
- `source/layer/details/softmax.cpp`
- `source/layer/details/relu.cpp`
- `include/utils/math/fmath.hpp`

**核心知识点**：

1. **ActivationLayer 统一调度**

   ```cpp
   class ActivationLayer : public NonParamLayer {
       ActivationType act_type_;
       
       Forward() {
           // 根据类型选择 SSE 优化的函数指针
           ActivationFunc func = ApplySSEActivation(act_type_);
           #pragma omp parallel for
           for (int i = 0; i < batch; i++) {
               func(inputs[i], outputs[i]);  // 调用 SIMD 实现
           }
       }
   };
   ```

   支持的激活函数：ReLU、SiLU、ReLU6、Sigmoid、HardSigmoid、HardSwish

2. **SIMD 向量化（Softmax 为例）**

   Softmax 使用 **AVX2（256 位 = 8 个 float 并行）**：

   ```cpp
   // 第一遍：计算 exp(x - max) 并累加
   __m256 max_val = _mm256_set1_ps(max);
   __m256 sum = _mm256_setzero_ps();
   for (...) {
       __m256 data = _mm256_loadu_ps(ptr);     // 加载 8 个 float
       __m256 diff = _mm256_sub_ps(data, max); // x - max
       __m256 exp_val = fmath::exp_ps256(diff); // SIMD 指数近似
       sum = _mm256_add_ps(sum, exp_val);       // 累加
       _mm256_storeu_ps(out, exp_val);          // 存储
   }
   // Horizontal reduce 求 sum 的总和
   // 第二遍：除以总和完成归一化
   ```

   **关键 SIMD 指令**：
   | 指令 | 作用 |
   |------|------|
   | `_mm256_loadu_ps` | 加载 8 个 float（非对齐地址） |
   | `_mm256_set1_ps` | 广播单个值到 8 个 lane |
   | `_mm256_sub_ps` | 逐元素减法 |
   | `fmath::exp_ps256` | SIMD 指数函数近似 |
   | `_mm256_hadd_ps` | 水平加法（用于归约求和） |

3. **fmath 库**

   项目集成了 [herumi/fmath](https://github.com/herumi/fmath/)，提供 SSE/AVX 版本的数学函数：
   - `exp_ps256` — SIMD 指数
   - `sqrt_ps256` — SIMD 开方
   - `reciprocal_ps256` — SIMD 倒数近似

**思考题**：
- 为什么 Softmax 要分两遍？（需要先求 sum 才能做归一化，无法单遍完成）
- SIMD 的加速比理论上是 8 倍（AVX2），实际能达到多少？为什么？（受内存带宽、分支预测等限制，通常 3-5 倍）

## 第三阶段：计算图与运行时（Day 8-11）

### Day 8：PNNX 中间表示

**目标**：理解模型文件格式和 IR（中间表示）的概念。

**阅读文件**：
- `include/runtime/pnnx/ir.h`
- `include/runtime/runtime_operand.hpp`
- `include/runtime/runtime_attr.hpp`
- `include/runtime/runtime_parameter.hpp`
- `include/runtime/runtime_datatype.hpp`

**核心知识点**：

1. **PNNX IR 层次**

   ```
   pnnx::Graph                       // 整张计算图
   ├── pnnx::Operator[]              // 所有算子
   │   ├── name                      // 算子实例名
   │   ├── type                      // 算子类型（"nn.Conv2d"）
   │   ├── inputs: pnnx::Operand[]   // 输入操作数
   │   ├── outputs: pnnx::Operand[]  // 输出操作数
   │   ├── parameters                // 标量参数（stride, padding...）
   │   └── attributes                // 张量属性（weight, bias...）
   └── ...
   
   pnnx::Operand                     // 数据节点
   ├── name                          // 名称
   ├── shapes                        // 形状
   └── type                          // 数据类型
   ```

2. **KuiperInfer 的对应结构**

   ```
   RuntimeGraph                     // 运行时图
   └── RuntimeOperator[]            // 运行时算子
       ├── input_operands           // 输入（含实际 Tensor 数据）
       ├── output_operators         // 下游算子映射
       ├── attribute                // 权重（RuntimeAttribute）
       ├── params                   // 参数（RuntimeParameter）
       └── layer                    // 具体算子实现（Layer<float>*）
   ```

3. **RuntimeAttribute 的内存优化**

   ```cpp
   // 读取权重后，立即释放原始数据
   std::vector<T> get<T>() {
       std::vector<T> result;
       // 从 weight_data 复制数据到 result
       weight_data.swap(empty_vec);  // 释放内存！权重只用一次
       return result;
   }
   ```

4. **RuntimeParameter 类型层次**

   ```
   RuntimeParameter                     // 基类
   ├── RuntimeParameterInt / IntArray
   ├── RuntimeParameterFloat / FloatArray
   ├── RuntimeParameterString / StringArray
   └── RuntimeParameterBool
   ```

   这是一个**类型对象模式**，用继承来存储异构参数。

**思考题**：
- PNNX 和 ONNX 的本质区别是什么？（PNNX 是 NCNN 的专用 IR，更紧凑；ONNX 是开放标准，更通用）

### Day 9：RuntimeGraph 初始化与图构建

**目标**：理解计算图的完整构建流程。

**阅读文件**：
- `include/runtime/runtime_ir.hpp`
- `source/runtime/runtime_ir.cpp`（重点读 `Init()` 和 `Build()`）
- `include/runtime/runtime_op.hpp`
- `source/runtime/runtime_op.cpp`

**核心知识点**：

1. **RuntimeGraph 状态机**

   ```
   NeedInit (-2) ──Init()──→ NeedBuild (-1) ──Build()──→ Complete (0)
   ```

2. **Init() 流程**

   ```
   加载 .param 和 .bin 文件
       ↓
   遍历 PNNX 图中的每个 Operator
       ↓
   为每个 Operator 创建 RuntimeOperator 对象
       ├── 复制 inputs（名称、形状、类型）
       ├── 复制 outputs
       ├── 提取 attributes（权重数据）
       └── 提取 parameters（标量参数）
   ```

3. **Build() 流程（核心！）**

   ```
   ① CreateNodeRelation()
      ├── 遍历所有算子，匹配 "输出名 → 输入名" 建立连接关系
      ├── 为每个非 Input/Output 的算子创建 Layer 实例（通过工厂）
      └── 将 Layer 绑定到 RuntimeOperator
   
   ② ReverseTopoSort()
      ├── 从每个未访问的算子开始 DFS
      ├── 通过后序遍历分配 start_time
      └── 按 start_time 排序 → 得到执行顺序
   
   ③ InitOperatorInput/Output()
      ├── 为所有算子的输入输出预分配 Tensor 内存
      └── 内存复用优化：不再需要的输出空间可被后续算子复用
   
   ④ 释放原始 PNNX 图（已不需要）
   ```

4. **RuntimeOperator 结构设计**

   ```cpp
   class RuntimeOperatorBase<T> {
       int start_time;                          // 执行顺序
       int end_time;                            // 最后消费者的执行时间
       bool has_forward;                        // 是否已执行过
       
       // 输入：生产者名 → 数据
       std::map<std::string, RuntimeOperand<T>> input_operands;
       
       // 输出：下游消费者名 → 消费者算子
       std::map<std::string, RuntimeOperatorBase<T>*> output_operators;
       
       // 权重和参数
       std::map<std::string, RuntimeAttribute> attribute;
       std::map<std::string, RuntimeParameterBase> params;
       
       // 具体算子实现
       std::shared_ptr<Layer<T>> layer;
   };
   ```

PNNX 图:

Build 后:
  [Op:Input, t=0] → [Op:Conv2d, t=1] → [Op:BatchNorm, t=2] → ...
                        ↓                     ↓
                   Layer:Conv           Layer:BatchNorm

### Day 10：拓扑排序详解

**目标**：深入理解计算图的执行顺序确定算法。

**阅读文件**：
- `source/runtime/runtime_ir.cpp` 中的 `ReverseTopoSort()` 和辅助函数

**核心知识点**：

1. **反向拓扑排序算法**

   ```cpp
   int ReverseTopoSort(RuntimeOperator& node, int count) {
       if (node.has_forward) return count;      // 已访问，返回
       node.has_forward = true;
       
       // 递归访问所有下游节点
       for (auto& [name, next_op] : node.output_operators) {
           count = ReverseTopoSort(*next_op, count);
       }
       
       // 后序：所有下游都访问完后，才给自己编号
       node.start_time = ++count;
       return count;
   }
   ```

   **关键**：后序遍历保证——一个节点编号比所有下游节点小，所以按编号升序执行就是正确的拓扑序。

2. **end_time 计算**

   ```cpp
   // end_time = 最后消费该算子输出的算子的 start_time
   // 用于内存复用判断：如果 end_time < 当前算子的 start_time，
   // 说明该算子的输出不再需要，空间可以被复用
   ```

3. **内存复用优化**

   ```cpp
   InitOperatorOutput() {
       for (each operator in execution order) {
           for (each output) {
               bool can_reuse = false;
               // 检查是否有之前算子的输出不再被需要
               for (each previous output) {
                   if (previous.end_time < current.start_time 
                       && size matches) {
                       can_reuse = true;
                       // 复用已有内存，不新分配
                   }
               }
               if (!can_reuse) allocate_new();
           }
       }
   }
   ```

手绘以下计算图的拓扑排序过程：

        Concat
          ↓

      /      \
 BatchNorm   SkipConnection
      \      /

          ↓
        ReLU
          ↓
       Output

### Day 11：前向执行与数据传播

**目标**：理解 Forward 的完整执行流程。

**阅读文件**：
- `source/runtime/runtime_ir.cpp` 中的 `Forward()` 和 `PropagateLayerOutputs()`

**核心知识点**：

1. **Forward 主循环**

   ```cpp
   StatusCode RuntimeGraph::Forward() {
       for (auto& op : operators_) {  // 按拓扑序遍历
           if (op 是 Input 或 Output) continue;
           
           // 执行算子
           op.layer->Forward();
           
           // 将输出传播到下游算子的输入空间
           PropagateLayerOutputs(op, outputs);
       }
   }
   ```

2. **PropagateLayerOutputs（零拷贝数据流）**

   ```cpp
   void PropagateLayerOutputs(RuntimeOperator& current, outputs) {
       for (auto& [consumer_name, consumer_op] : current.output_operators) {
           // 将当前算子的输出 shared_ptr 赋值给消费者的输入 slot
           // 不拷贝数据！只是传递指针
           consumer_op.input_operands[current.name].datas = outputs;
       }
   }
   ```

   **为什么零拷贝？** 因为 `shared_ptr<Tensor>` 只是引用计数 + 指针，传递代价极小。实际数据在原地。

3. **完整数据流**

   ```
   用户输入图像
       ↓
   PreProcessImage() → Tensor[3, 224, 224]
       ↓
   graph.set_inputs("pnnx_input_0", inputs)
       ↓ 传播到 Input 算子
   graph.Forward()
       ↓
   [Conv2d] 读取 input_operands → 计算 → 产生 outputs
       ↓ PropagateLayerOutputs
   [BatchNorm] 读取 input_operands → 计算 → 产生 outputs
       ↓ PropagateLayerOutputs
   [ReLU] → ... → [Output]
       ↓
   graph.get_outputs("pnnx_output_0") → 原始 logits
       ↓
   用户后处理（Softmax、取 argmax）
   ```

**思考题**：
- 如果两个算子共享同一个输入 Tensor，零拷贝会有什么影响？（`shared_ptr` 的引用计数 +1，数据不会被提前释放）
- Forward 中可以插入算子级计时吗？（可以，每个算子有 `start_time`/`end_time`，Debug 模式下有 `LayerTimeLogging`）

## 第四阶段：表达式解析器（Day 12-13）

### Day 12：词法分析与语法分析

**目标**：理解 PNNX 表达式语言的解析过程。

**阅读文件**：
- `include/parser/parse_expression.hpp`
- `source/parser/parse_expression.cpp`

**核心知识点**：

1. **为什么要解析表达式？**

   PNNX 模型中某些算子的输出形状不是固定的，而是用表达式定义的：
   ```
   输出形状 = mul(@0, add(@1, 1))
   ```
   其中 `@0`、`@1` 引用输入张量的某个维度。需要在运行时求值才能得到真实形状。

2. **词法分析（Tokenizer）**

   ```
   输入: "mul(@0, add(@1, @2))"
       ↓
   移除空白字符
       ↓
   扫描 Token:
   [OP:"mul"] [BRACKET:"("] [INPUT:"@0"] [COMMA:","] 
   [OP:"add"] [BRACKET:"("] [INPUT:"@1"] [COMMA:","] 
   [INPUT:"@2"] [BRACKET:")"] [BRACKET:")"]
   ```

   Token 类型：
   - `TokenOpMul` / `TokenOpAdd` — 运算操作
   - `TokenInputNumber` — 输入引用（`@N`）
   - `TokenBracket` — 括号
   - `TokenComma` — 逗号

3. **语法分析（递归下降解析）**

   ```cpp
   TokenNode* Generate() {
       // 解析运算操作
       if (当前是 "add" 或 "mul") {
           auto op_node = CreateOpNode();
           consume("(");
           op_node->children.push_back(Generate());  // 递归解析第一个子表达式
           while (遇到 ",") {
               consume(",");
               op_node->children.push_back(Generate());
           }
           consume(")");
           return op_node;
       }
       // 解析叶子节点（输入引用）
       return CreateInputNode();
   }
   ```

   生成的语法树（AST）：
   ```
         mul
        /   \
      @0    add
           /  \
         @1   @2
   ```

### Day 13：表达式求值与 Expression 算子

**目标**：理解 AST 求值和 Expression 算子的实现。

**阅读文件**：
- `source/layer/details/expression.cpp`

**核心知识点**：

1. **逆波兰序（RPN）求值**

   ```
   AST:     mul              RPN:  @0 @1 @2 add mul
           /   \             (后序遍历)
         @0    add
              /  \
            @1   @2
   ```

2. **栈机器求值**

   ```cpp
   // 逆序遍历 RPN token 列表
   for (auto rit = tokens.rbegin(); rit != tokens.rend(); ++rit) {
       if (是输入引用) {
           stack.push(对应输入的 Tensor);
       } else if (是 Add 操作) {
           auto b = stack.top(); stack.pop();
           auto a = stack.top(); stack.pop();
           stack.push(TensorElementAdd(a, b));
       } else if (是 Mul 操作) {
           auto b = stack.top(); stack.pop();
           auto a = stack.top(); stack.pop();
           stack.push(TensorElementMultiply(a, b));
       }
   }
   // 最终栈上只剩一个结果
   ```

3. **Expression 算子的作用**

   在 PNNX 中，某些融合操作被表示为 Expression 算子：
   ```
   # 例如 LayerNorm 可以表示为:
   output = (input - mean) / sqrt(var + eps) * weight + bias
   # 在 PNNX 中可能被编码为:
   pnnx.Expression: "add(mul(sub(input0, input1), input2), input3)"
   ```

**思考题**：
- 为什么逆序遍历 RPN 就能正确求值？（RPN 是后序遍历，操作符在最后；逆序后先遇到操作数，push 到栈，遇到操作符时栈顶刚好是它的操作数）
- Expression 算子支持哪些运算？目前只有 `+` 和 `*`，够用了吗？

## 第六阶段：高性能推理进阶（Day 16-20）

### Day 16：im2col + GEMM 深度优化

**目标**：彻底掌握卷积加速的 im2col 技术。

**延伸阅读与实践**：

1. **im2col 的内存布局**

   ```
   输入: [C=3, H=5, W=5]  卷积核: [2, 3, 3, 3]  stride=1  pad=1
   输出: [2, 5, 5]
   
   im2col 结果: [C*kh*kw=27, H_out*W_out=25]
   每列 = 一个输出位置对应的感受野展平 (3*3*3 = 27 个值)
   
   卷积核矩阵: [out_ch=2, C*kh*kw=27]
   
   GEMM: [2, 27] × [27, 25] → [2, 25] → reshape → [2, 5, 5]
   ```

2. **不同卷积模式的 im2col**

   | 模式 | 特点 |
   |------|------|
   | 1×1 conv, no pad | 无需 im2col，直接矩阵乘法 |
   | 标准 conv | 完整 im2col + GEMM |
   | Depthwise conv | 逐 channel 做 im2col |
   | Group conv | 按 group 分割后分别做 im2col |

3. **对比：直接卷积 vs im2col**

   ```
   直接卷积: O(C_in * kh * kw * H_out * W_out * C_out) 次操作
             内存访问模式不规则，缓存不友好
             
   im2col:   一次 O(C*kh*kw*H_out*W_out) 的数据拷贝
             + 一次高度优化的 GEMM
             内存访问连续，充分利用 CPU 缓存和 SIMD
   ```

### Day 17：SIMD 向量化编程

**目标**：掌握 SIMD 指令在深度学习推理中的应用，从零实现 6 种激活函数的 AVX2 向量化版本。

**阅读文件**：
- `learning/day17/main.cpp` — 完整 SIMD 教程代码
- `source/layer/details/simd.cpp` — 生产代码 SIMD 实现
- `include/utils/math/fmath.hpp` — SIMD 数学函数库

**核心知识点**：

1. **SIMD 指令层次**

   | 指令集 | 位宽 | float 并行数 | CPU 支持 |
   |--------|------|-------------|----------|
   | SSE2   | 128  | 4           | 几乎所有 x86 |
   | AVX    | 256  | 8           | Intel Sandy Bridge+ |
   | AVX2   | 256  | 8 (+整数)   | Intel Haswell+ |
   | AVX-512| 512  | 16          | Intel Skylake-X+ |

2. **SIMD 编程三板斧**

   ```cpp
   // ① LOAD: 从内存加载数据到向量寄存器
   __m256 va = _mm256_loadu_ps(&a[i]);     // 加载 8 个 float（非对齐）
   __m256 vb = _mm256_set1_ps(val);        // 广播标量到 8 个 lane
   
   // ② COMPUTE: 一条指令，多组数据
   __m256 vc = _mm256_add_ps(va, vb);      // 8 次加法一次完成
   __m256 vc = _mm256_mul_ps(va, vb);      // 8 次乘法一次完成
   __m256 vc = _mm256_max_ps(zero, va);    // 8 次比较（ReLU）
   
   // ③ STORE: 写回内存
   _mm256_storeu_ps(&c[i], vc);            // 存储 8 个 float
   
   // Tail 处理（不足 8 个的元素）
   for (; i < n; i++) c[i] = a[i] + b[i];
   ```

3. **SIMD exp 近似（Sigmoid 核心）**

   生产代码中的 Sigmoid/SiLU 使用 `fmath::exp_ps256`（查表法 + 多项式）。
   Day 17 教学代码用 6 项 Taylor 级数演示原理：

   ```cpp
   // Range reduction: x = i*ln2 + f
   // exp(f) ≈ 1 + f*(1 + f*(1/2 + f*(1/6 + f*(1/24 + f/120))))
   // 2^i via IEEE 754 指数位操作
   // exp(x) = exp(f) * 2^i
   ```

   精度 ~1e-6，速度比 `std::exp` 快 **10x**。

4. **SIMD 条件选择（Blend / Mask）**

   用比较指令生成 mask，再用 blend 做分支选择，消除 if-else 的分支预测开销：

   ```cpp
   // HardSwish: 三路条件选择
   __m256 ge_mask = _mm256_cmp_ps(x, three, _CMP_GE_OS);   // x >= 3
   __m256 mid_mask = ...;                                    // -3 < x < 3
   __m256 result = _mm256_add_ps(
       _mm256_and_ps(f_high, ge_mask),
       _mm256_and_ps(f_mid, mid_mask));
   ```

   `_mm256_blendv_ps` 是更简洁的 blend 指令：
   ```cpp
   // mask=1 → 选 a, mask=0 → 选 b
   result = _mm256_blendv_ps(b, a, mask);
   ```

5. **SIMD MatMul 向量化**

   MatMul 的 SIMD 向量化一次计算 8 个输出元素：
   ```cpp
   for (j = 0; j + 8 <= N; j += 8) {
       __m256 acc = _mm256_setzero_ps();
       for (k = 0; k < K; k++) {
           acc = _mm256_fmadd_ps(_mm256_set1_ps(A[i][k]),
                                  _mm256_loadu_ps(B[k][j..j+7]), acc);
       }
       _mm256_storeu_ps(C[i][j..j+7], acc);
   }
   ```

6. **6 种激活函数的 SIMD 实现**

   | 激活函数 | SIMD 加速 | 技术要点 |
   |----------|----------|----------|
   | ReLU | ~1x | `_mm256_max_ps`，编译器可自动向量化 |
   | ReLU6 | ~1x | `_mm256_min_ps(_mm256_max_ps(...))` |
   | Sigmoid | **10x** | Taylor 级数 exp 近似 |
   | HardSwish | **13x** | 三路 blend mask |
   | SiLU | **10x** | exp 近似 + `_mm256_blendv_ps` |
   | HardSigmoid | **12x** | 三路 blend mask |

7. **手写 SIMD 的三大高价值场景**

   - **transcendental 函数**（exp, log, sin）— 编译器不会自动替换多项式近似
   - **条件分支**（if-else）— 编译器很难将分支转为 blend mask
   - **GEMM / MatMul** — 编译器对嵌套循环的向量化有限

### Day 18：OpenMP 多线程并行策略 — SIMD + 多核全面加速

**目标**：在 Day 17 SIMD 的基础上叠加 OpenMP 多线程，实现"双层加速"（SIMD 向量化 × 多核并行）。

**阅读文件**：
- `learning/day18/main.cpp` — 完整 OpenMP + SIMD 教程代码
- `source/layer/details/activation.cpp` — 生产代码 ActivationLayer（OpenMP + SIMD 组合）
- `source/layer/details/base_convolution.cpp` — 生产代码 Conv（batch × group 嵌套并行）

**核心知识点**：

1. **双层加速模式**

   ```
   Day 17: SIMD (AVX2) → 单核内 8 路并行 → 5-13x 加速
   Day 18: OpenMP      → 多核 T 路并行  → T 倍加速
   组合:    SIMD + OMP  → 8 × T 理论上限 → 20-34x 加速
   ```

   ```cpp
   // 外层: OpenMP 并行（多核）
   #pragma omp parallel for num_threads(T)
   for (int t = 0; t < T; t++) {
       // 每线程处理 N/T 个元素
       // 内层: SIMD 向量化（单核内 8 路并行）
       for (; i + 8 <= local_n; i += 8) {
           __m256 v = _mm256_loadu_ps(data + i);
           v = simd_compute(v);
           _mm256_storeu_ps(out + i, v);
       }
   }
   ```

2. **生产代码中的并行模式**

   | 层类型 | 并行策略 | 代码位置 |
   |--------|---------|---------|
   | ActivationLayer | batch 级 OpenMP + SIMD 内核 | `activation.cpp` |
   | ConvolutionLayer | batch × group 嵌套 OpenMP + GEMM | `base_convolution.cpp` |
   | SoftmaxLayer | batch 级 OpenMP + AVX2 内核 | `softmax.cpp` |
   | MaxPooling | batch 级 OpenMP + 标量循环 | `maxpooling.cpp` |

3. **激活函数的双层加速**

   ```cpp
   // OmpSimdActiv: 将数据拆分为 T 个 chunk，每个 chunk 运行 SIMD
   void OmpSimdActiv(const float* in, float* out, uint64_t n,
                     ActivationType type, int num_threads) {
   #pragma omp parallel for num_threads(num_threads)
     for (int64_t t = 0; t < num_threads; t++) {
         int64_t chunk = n / num_threads;
         int64_t start = t * chunk + min(t, n % num_threads);
         int64_t end = start + chunk + (t < n % num_threads);
         SimdActiv(in + start, out + start, end - start, type);
     }
   }
   ```

4. **MatMul 的 OpenMP 并行**

   将 M 维度（输出行数）拆分到线程，每行用 SIMD8 计算：

   ```cpp
   #pragma omp parallel for num_threads(T)
   for (int i = 0; i < M; i++) {
       // 每行独立计算，无数据依赖，天然线程安全
       for (j = 0; j + 8 <= N; j += 8) {
           __m256 acc = _mm256_setzero_ps();
           for (k = 0; k < K; k++) {
               acc += A[i][k] * B[k][j..j+7];
           }
           C[i][j..j+7] = acc;
       }
   }
   ```

   **性能数据**：
   | 场景 | Scalar | SIMD8 | SIMD+OMP | 总加速 |
   |------|--------|-------|----------|--------|
   | Conv GEMM [128,576]×[576,784] | 50ms | 7ms | 1.5ms | **34x** |
   | FC [512,1024]×[1024,512] | 698ms | 131ms | 76ms | **9x** |

5. **并行三原则**

   - **数据并行 > 任务并行** — 每个线程处理不同数据，无共享状态
   - **减少同步** — 能不用 reduction 就不用，避免 `#pragma omp critical`
   - **线程粒度** — 每线程至少处理 4KB 数据（否则线程开销占主导）

6. **四大并行陷阱**

   | 陷阱 | 原因 | 解决方案 |
   |------|------|---------|
   | 数据竞争 | 多线程写同一位置 | `reduction(+:)` 或分片写 |
   | False sharing | 不同线程写相邻缓存行 | `alignas(64)` 或 padding |
   | 过度并行 | 线程数 > 数据量 | 判断阈值后退化单线程 |
   | 负载不均衡 | 某些线程早完成 | `schedule(dynamic)` |

**关键性能数据**：

### Day 19：内存优化与算子融合

**目标**：理解推理框架的内存管理策略和算子融合思想。

**核心知识点**：

1. **内存复用机制**

   ```
   算子 A (t=1) → 算子 B (t=5)
   算子 C (t=3)
   
   算子 A 的输出在 t=5 时被 B 消费
   t=5 之后，A 的输出不再需要
   → 算子 C (t=3) 或后续算子可以复用 A 的输出内存
   ```

   判断条件：`output_occur_end_time < current_op.start_time`

2. **零拷贝数据传播**

   `shared_ptr<Tensor>` 传递 = 零拷贝。两个算子"共享"同一块内存，引用计数自动管理。

3. **算子融合思想**

   ```
   未融合: Conv → [中间激活值存内存] → BN → [中间激活值存内存] → ReLU
            ↑ 多次读写内存，带宽瓶颈
             
   融合:   Conv_BN_ReLU → 一次计算，中间值留在寄存器
            ↑ 减少内存读写，大幅提升速度
   ```

   KuiperInfer 目前没有深度算子融合，但 Expression 算子就是一种轻量级融合——把多个元素级操作融合在一个算子中执行。

4. **推理框架的内存布局优化**

   | 技术 | 说明 |
   |------|------|
   | NCHWc | 将 channel 维度分块，提高缓存局部性 |
   | NC8c / NC4c | NCNN 的优化布局，channel 按 8/4 分组 |
   | 内存池 | 预分配大块内存，避免运行时频繁 malloc/free |

**思考题**：
- 算子融合的收益是什么？代价是什么？（减少内存读写 vs 增加实现复杂度）
- 为什么推理时 BN 可以融合到 Conv 中？（推理时 BN 的参数是固定的，可以预计算合并到 Conv 的权重和偏置中）
